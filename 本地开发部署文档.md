# Check CX 本地开发部署文档

> Check CX 是一个基于 **Next.js 16** 的 AI 模型健康监控面板，前后端一体（Next.js App Router + API Routes + 后台轮询）。本文档说明如何在本地从 0 到 1 启动开发环境。

---

## 一、环境要求

| 工具 | 版本要求 | 说明 |
| ---- | -------- | ---- |
| Node.js | **>= 20.x** | 推荐 LTS 版本 |
| pnpm | **>= 10.10.0** | 项目使用 pnpm 作为包管理器 |
| Git | 任意现代版本 | 拉取代码 |
| Supabase 账号 | — | 用作数据库（PostgreSQL） |

### 1.1 安装 Node.js

前往 [https://nodejs.org/](https://nodejs.org/) 下载并安装 Node.js 20+ LTS。

校验：

```bash
node -v   # v20.x.x 或更高
```

### 1.2 安装 pnpm

```bash
# 方式一：使用 npm 全局安装
npm install -g pnpm

# 方式二：使用 corepack（Node 自带，推荐）
corepack enable
corepack prepare pnpm@10.10.0 --activate
```

校验：

```bash
pnpm -v   # 10.10.0 或更高
```

## 三、配置 Supabase 数据库

Check CX 把所有 Provider 配置、模型、模板、历史记录、分组、通知都保存在 Supabase 中，**必须先准备好数据库才能启动**。

### 3.1 创建 Supabase 项目

1. 访问 [https://supabase.com/](https://supabase.com/)，注册并新建一个项目。
2. 在项目 **Settings → API** 页面记录以下三个值：
   - `Project URL` → 对应 `SUPABASE_URL`
   - `anon public` key → 对应 `SUPABASE_PUBLISHABLE_OR_ANON_KEY`
   - `service_role secret` key → 对应 `SUPABASE_SERVICE_ROLE_KEY` ⚠️ 严禁泄漏

### 3.2 执行数据库迁移

项目内置了 SQL 迁移文件，位于 `supabase/migrations/` 目录。请按 **文件名时间戳从早到晚** 的顺序，在 Supabase 控制台的 **SQL Editor** 中依次执行：

```
supabase/migrations/
├── 20251204214400_create_group_info_table.sql
├── 20251204223300_enable_rls_group_info.sql
├── 20251208143800_add_validation_failed_status.sql
├── 20251208232000_create_system_notifications.sql
├── 20251210120000_add_error_status.sql
├── 20260113222614_extend_history_retention.sql
├── 20260114220000_add_group_tags.sql
├── 20260114220000_add_poller_leases.sql
├── 20260119000000_optimize_trend_query.sql
├── 20260120153000_add_history_config_index.sql
├── 20260123024500_drop_trend_history_rpc.sql
├── 20260304120000_add_request_templates.sql
├── 20260304153000_backfill_check_configs_to_templates.sql
├── 20260322120000_extract_check_models.sql
├── 20260322130000_relink_templates_through_models.sql
└── 20260325110000_include_degraded_in_availability_stats.sql
```

> 文件名带 `_dev.sql` 后缀的是开发环境专用版本，**两者二选一**：本地开发建议使用 `_dev.sql` 版本（若存在同名版本）。

执行完毕后，应该在 **Table Editor** 看到以下核心表：

- `check_models`：模型表
- `check_configs`：检查配置表
- `check_request_templates`：请求模板表
- `check_history`：历史记录表
- `group_info`：分组信息表
- `system_notifications`：系统通知表
- `check_poller_leases`：轮询器租约表

### 3.3 插入初始检查配置（可选）

新数据库是空的，启动后 Dashboard 不会显示任何卡片。可在 SQL Editor 中插入一条测试配置：

```sql
-- 1. 模板
INSERT INTO check_request_templates (name, type, request_header, metadata)
VALUES (
  'openai-default',
  'openai',
  '{"User-Agent": "check-cx"}',
  '{"temperature": 0}'
)
ON CONFLICT (name) DO NOTHING;

-- 2. 模型
INSERT INTO check_models (type, model, template_id)
SELECT 'openai', 'gpt-4o-mini', id
FROM check_request_templates
WHERE name = 'openai-default'
ON CONFLICT (type, model) DO NOTHING;

-- 3. 配置（替换 api_key 为你自己的）
INSERT INTO check_configs (name, type, model_id, endpoint, api_key, enabled)
SELECT '主力 OpenAI', 'openai', id,
       'https://api.openai.com/v1/chat/completions',
       'sk-xxxxxxxxxxxxxxxx', true
FROM check_models
WHERE type = 'openai' AND model = 'gpt-4o-mini';
```

## 四、配置环境变量

### 4.1 复制环境变量模板

```bash
# Windows PowerShell
copy .env.example .env.local

# macOS / Linux / Git Bash
cp .env.example .env.local
```

### 4.2 编辑 `.env.local`

打开 `.env.local`，填写实际值：

```ini
# === Supabase（必填） ===
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_PUBLISHABLE_OR_ANON_KEY=eyJhbGciOiJIUzI1Ni...      # anon key
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1Ni...             # service role key（仅服务端使用）

# === 运行模式 ===
NODE_ENV=dev

# === 轮询配置（可选，有默认值） ===
CHECK_POLL_INTERVAL_SECONDS=60         # 检测间隔，15~600 秒
CHECK_NODE_ID=local                    # 多节点部署时的节点标识
HISTORY_RETENTION_DAYS=30              # 历史保留天数 7~365
OFFICIAL_STATUS_CHECK_INTERVAL_SECONDS=60
CHECK_CONCURRENCY=8                    # 最大并发数
```

> ⚠️ **安全提示**：`SUPABASE_SERVICE_ROLE_KEY` 拥有绕过 RLS 的权限，**绝对不要** 提交到 Git，也不要暴露到前端代码。

## 五、安装依赖

在项目根目录执行：

```bash
pnpm install
```

首次安装会下载约 300~500MB 的依赖，耐心等待。出现以下日志即代表完成：

```
Done in xx.xs using pnpm v10.x.x
```

如果 `pnpm install` 报权限错误（Windows 常见），尝试以管理员身份运行终端，或删除 `node_modules` 后重试。

## 六、启动开发服务器

### 6.1 启动命令

```bash
pnpm dev
```

控制台输出大致如下：

```
 ▲ Next.js 16.2.1
 - Local:        http://localhost:3000
 - Environments: .env.local

 ✓ Ready in 2.5s
```

### 6.2 验证启动

打开浏览器访问：

```
http://localhost:3000
```

应该看到 Dashboard 页面。如果数据库中已经有 `check_configs` 配置项，几秒后会出现首批检测结果卡片。

### 6.3 前后端说明

Check CX 是一个 **Next.js 全栈项目**，前后端共用同一个进程，启动 `pnpm dev` 后包含：

| 模块 | 路径 | 说明 |
| ---- | ---- | ---- |
| 前端 UI | `app/`、`components/` | React Server Components + Client Components |
| API 路由（后端） | `app/api/*` | 例：`/api/dashboard`、`/api/v1/status` |
| 后台轮询器 | `lib/core/poller.ts` | **模块加载时自动启动**，按 `CHECK_POLL_INTERVAL_SECONDS` 周期检测 |
| 官方状态轮询 | `lib/core/official-status-poller.ts` | 定时抓取 OpenAI/Anthropic 官方状态 |

> 你 **不需要** 单独启动后端服务，`pnpm dev` 一条命令即可启动完整的前后端 + 后台任务。

如果不想配置本地 Node 环境，可以直接用 Docker 跑预构建镜像。

## 九、调试与排错

### 9.1 查看轮询器日志

`pnpm dev` 模式下，所有轮询日志会输出到终端，包括：

- 每次检测开始/结束时间
- 每个配置的状态（operational/degraded/failed/maintenance）
- 首 token 延迟、Ping 延迟
- 历史写入结果
- 下次预计执行时间

### 9.2 常见问题

| 现象 | 原因 | 解决 |
| ---- | ---- | ---- |
| 启动后 Dashboard 空白 | 数据库无 `check_configs` 记录 | 参考 §3.3 插入测试配置 |
| 启动报错 `Missing SUPABASE_URL` | `.env.local` 未配置或未生效 | 确认文件在项目根目录，重启 dev server |
| 卡片一直显示 failed | API Key 错误、网络不通、endpoint 错误 | 在 Supabase 中修正 `api_key` 与 `endpoint` |
| 端口 3000 被占用 | 其它进程占用 | `pnpm dev -- -p 3001` 改端口 |
| 轮询不执行 | 多节点选主未抢到租约 | 单机开发可清空 `check_poller_leases` 表 |
| 热重载后轮询重复 | 已通过 `global-state.ts` 防御 | 若仍重复，重启 dev server |

### 9.3 重置数据库（开发环境）

如果想清空所有检测历史重新开始：

```sql
TRUNCATE TABLE check_history;
TRUNCATE TABLE check_poller_leases;
```
