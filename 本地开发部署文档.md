# Check CX 本地开发部署文档

> Check CX 是一个基于 **Next.js** 的 AI 模型健康监控面板，前后端一体（Next.js App Router + API Routes + 后台轮询）。数据存储使用本地 **SQLite**（better-sqlite3），首次启动时自动建表，无需外部数据库服务。本文档说明如何在本地从 0 到 1 启动开发环境，以及如何用 Docker 部署。

---

## 一、环境要求

| 工具 | 版本要求 | 说明 |
| ---- | -------- | ---- |
| Node.js | **>= 20.x** | 推荐 LTS 版本 |
| pnpm | **>= 10.10.0** | 项目使用 pnpm 作为包管理器 |
| Git | 任意现代版本 | 拉取代码 |
| sqlite3 CLI | 任意版本（可选） | 检查/编辑本地数据库，非必须 |

### 1.1 安装 Node.js

前往 [https://nodejs.org/](https://nodejs.org/) 下载并安装 Node.js 20+ LTS。

校验：

```bash
node -v   # v20.x.x 或更高
```

### 1.2 安装 pnpm

```bash
# 方式一：使用 npm 全局安装
npm install -g pnpm

# 方式二：使用 corepack（Node 自带，推荐）
corepack enable
corepack prepare pnpm@10.10.0 --activate
```

校验：

```bash
pnpm -v   # 10.10.0 或更高
```

## 三、数据库说明

Check CX 使用 **本地 SQLite 文件**（`better-sqlite3`）存储所有数据，**无需任何外部数据库服务**。

- 数据库文件路径由环境变量 `SQLITE_DB_PATH` 指定，默认为 `./data/monitor.db`。
- 首次启动时，应用会自动调用 `runMigrations()` 读取 `lib/db/schema.sql` 并建表，操作是**幂等的**，重复运行不会破坏数据。
- `./data/` 目录不存在时会自动创建。

**无需**手动执行任何迁移命令，也无需连接云端数据库。

### 3.1 核心数据表

首次启动后，`data/monitor.db` 中会包含以下核心表：

- `check_models`：模型表
- `check_configs`：检查配置表
- `check_request_templates`：请求模板表
- `check_history`：历史记录表
- `group_info`：分组信息表
- `system_notifications`：系统通知表

### 3.2 插入初始检查配置

新数据库是空的，启动后 Dashboard 不会显示任何卡片。推荐通过 **后台管理界面**（`/admin`）添加配置（见第六节）。

也可以用 sqlite3 CLI 直接插入。注意 SQLite 中 id 需由应用生成的小写十六进制 UUID 填写，时间戳为 ISO8601 TEXT，**不能使用** `gen_random_uuid()` 或 `now()`：

```bash
sqlite3 data/monitor.db
```

## 四、配置环境变量

### 4.1 复制环境变量模板

```bash
# Windows PowerShell
copy .env.example .env

# macOS / Linux / Git Bash
cp .env.example .env
```

### 4.2 编辑 `.env`

打开 `.env`，按需填写以下变量：

```ini
# === 认证（必填） ===
ADMIN_LOGIN_KEY=your-admin-password          # 后台登录密码
ADMIN_SESSION_SECRET=your-32-byte-hex-secret # 会话加密密钥，至少 32 字节

# === 数据库（可选，有默认值） ===
SQLITE_DB_PATH=./data/monitor.db             # SQLite 文件路径，默认 ./data/monitor.db

# === 应用地址（可选） ===
APP_URL=http://localhost:3000

# === 轮询配置（可选，有默认值） ===
CHECK_POLL_INTERVAL_SECONDS=60               # 检测间隔，15~600 秒，默认 60
HISTORY_RETENTION_DAYS=30                    # 历史保留天数
OFFICIAL_STATUS_CHECK_INTERVAL_SECONDS=60   # 官方状态轮询间隔
CHECK_CONCURRENCY=8                          # 最大并发数
```

生成安全的 `ADMIN_SESSION_SECRET`（至少 32 字节）：

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

> ⚠️ **安全提示**：`.env` 文件包含敏感密钥，**绝对不要**提交到 Git。

#### 完整环境变量一览

| 变量名 | 是否必填 | 默认值 | 说明 |
| ------ | -------- | ------ | ---- |
| `ADMIN_LOGIN_KEY` | **必填** | — | 后台登录密码 |
| `ADMIN_SESSION_SECRET` | **必填** | — | 会话签名密钥，≥32 字节 |
| `SQLITE_DB_PATH` | 可选 | `./data/monitor.db` | SQLite 文件路径 |
| `APP_URL` | 可选 | — | 应用外部访问地址 |
| `CHECK_POLL_INTERVAL_SECONDS` | 可选 | `60` | 检测间隔（15~600 秒） |
| `HISTORY_RETENTION_DAYS` | 可选 | — | 历史记录保留天数 |
| `OFFICIAL_STATUS_CHECK_INTERVAL_SECONDS` | 可选 | — | 官方状态页轮询间隔 |
| `CHECK_CONCURRENCY` | 可选 | — | 最大并发检测数 |

## 五、安装依赖

在项目根目录执行：

```bash
pnpm install
```

首次安装会下载约 300~500MB 的依赖（含 `better-sqlite3` 原生模块的预编译包），耐心等待。出现以下日志即代表完成：

```
Done in xx.xs using pnpm v10.x.x
```

如果 `pnpm install` 报权限错误（Windows 常见），尝试以管理员身份运行终端，或删除 `node_modules` 后重试。

## 六、启动开发服务器

### 6.1 启动命令

```bash
pnpm dev
```

控制台输出大致如下：

```
 ▲ Next.js 16.x.x
 - Local:        http://localhost:3000
 - Environments: .env

 ✓ Ready in 2.5s
```

首次启动时，应用会自动在 `./data/monitor.db` 创建数据库并建表，无需任何手动操作。

### 6.2 验证启动

打开浏览器访问：

```
http://localhost:3000          # 公开 Dashboard（只读）
http://localhost:3000/admin    # 后台管理界面（需登录）
```

- **公开面板**：`/`、`/group/[groupName]`，以及只读 API `/api/v1/status` 等，无需登录。
- **后台管理**：访问 `/admin/login`，输入 `ADMIN_LOGIN_KEY` 登录。可在此添加/编辑 Provider 配置。

如果数据库中已经有 `check_configs` 配置项，几秒后会出现首批检测结果卡片。

### 6.3 前后端说明

Check CX 是一个 **Next.js 全栈项目**，前后端共用同一个进程，启动 `pnpm dev` 后包含：

| 模块 | 路径 | 说明 |
| ---- | ---- | ---- |
| 公开前端 UI | `app/`、`components/` | Dashboard、分组页，React Server Components + Client Components |
| 后台管理 UI | `app/admin/` | 配置管理，需登录 |
| API 路由（后端） | `app/api/*` | 例：`/api/dashboard`、`/api/v1/status` |
| 后台轮询器 | `lib/core/poller.ts` | **模块加载时自动启动**，按 `CHECK_POLL_INTERVAL_SECONDS` 周期检测 |
| 官方状态轮询 | `lib/core/official-status-poller.ts` | 定时抓取 OpenAI/Anthropic 官方状态 |

> 你 **不需要** 单独启动后端服务，`pnpm dev` 一条命令即可启动完整的前后端 + 后台任务。

## 八、Docker 部署

Docker 镜像在构建阶段原生编译 `better-sqlite3`（build stage 含 python3/make/g++），standalone 镜像携带 `lib/db/schema.sql` 和原生模块，`data/` 目录挂载为外部卷持久化数据。

> ⚠️ **单进程限制**：SQLite 同一时间只能由一个进程写入。**不要**同时运行多个容器实例指向同一 `data/` 卷，否则会数据损坏。

### 8.1 准备 `.env` 文件

```bash
cp .env.example .env
# 编辑 .env，至少填写 ADMIN_LOGIN_KEY 和 ADMIN_SESSION_SECRET
```

### 8.2 启动容器

```bash
# 方式一：使用 deploy.sh（Linux/macOS/Git Bash）
./deploy.sh

# 方式二：直接 docker compose
docker compose up -d
```

`docker-compose.yml` 核心配置：

```yaml
services:
  monitor:
    build: .
    ports:
      - "3000:3000"
    env_file: .env
    volumes:
      - ./data:/app/data    # SQLite 数据持久化
    restart: unless-stopped
```

启动后访问 `http://localhost:3000`。

### 8.3 查看容器日志

```bash
docker logs check-cx --tail 100 -f
```

### 8.4 停止容器

```bash
docker compose down
```

### 8.5 数据备份

数据全部在 `./data/` 目录中，备份即复制该目录，或使用 sqlite3 备份命令：

```bash
sqlite3 data/monitor.db ".backup data/monitor.db.bak"
```

## 九、调试与排错

### 9.1 查看轮询器日志

`pnpm dev` 模式下，所有轮询日志会输出到终端，包括：

- 每次检测开始/结束时间
- 每个配置的状态（operational/degraded/failed/maintenance）
- 首 token 延迟、Ping 延迟
- 历史写入结果
- 下次预计执行时间

### 9.2 常见问题

| 现象 | 原因 | 解决 |
| ---- | ---- | ---- |
| 启动后 Dashboard 空白 | 数据库无 `check_configs` 记录 | 登录 `/admin` 添加配置，或参考 §3.2 手动插入 |
| 启动报错 `Missing ADMIN_LOGIN_KEY` | `.env` 未配置或未生效 | 确认 `.env` 在项目根目录，重启 dev server |
| 启动报错 `ADMIN_SESSION_SECRET too short` | 密钥长度不足 32 字节 | 用 `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` 重新生成 |
| 卡片一直显示 failed | API Key 错误、网络不通、endpoint 错误 | 登录 `/admin` 修正对应配置的 `api_key` 与 `endpoint` |
| 端口 3000 被占用 | 其它进程占用 | `pnpm dev -- -p 3001` 改端口 |
| 热重载后轮询重复 | 已通过 `global-state.ts` 防御 | 若仍重复，重启 dev server |
| `better-sqlite3` 安装失败 | 原生模块需要编译工具 | Windows 需安装 `windows-build-tools`；Linux/macOS 需 python3/make/g++ |

### 9.3 重置数据库（开发环境）

如果想清空所有检测历史重新开始：

```bash
sqlite3 data/monitor.db "DELETE FROM check_history;"
```

如果想完全重置（删除数据库文件，下次启动自动重建）：

```bash
# Windows PowerShell
Remove-Item data\monitor.db

# macOS / Linux
rm data/monitor.db
```
